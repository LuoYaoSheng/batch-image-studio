# Batch Image Studio - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Batch Image Studio, decomposing the requirements from the PRD, UX Design, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

**图片导入 (FR1-FR5):**
- FR1: 用户可以导入单个或多个图片文件
- FR2: 用户可以导入包含图片的文件夹
- FR3: 系统显示已导入图片的缩略图列表
- FR4: 用户可以从已导入列表中移除单张图片
- FR5: 系统自动选择第一张图片作为样图

**区域框选 (FR6-FR13):**
- FR6: 系统在图片上显示一个默认的处理区域框
- FR7: 用户可以拖动区域框改变位置
- FR8: 用户可以拖拽区域框的手柄调整大小
- FR9: 用户可以旋转区域框
- FR10: 用户可以清除当前区域框
- FR11: 用户可以重新进入框选模式替换当前区域
- FR12: 系统支持锚定模式（区域固定在图片的某个角）
- FR13: 系统支持相对坐标模式（区域按图片比例缩放）

**预览生成 (FR14-FR16):**
- FR14: 用户可以生成当前区域和处理方式的预览效果
- FR15: 系统显示处理前后的对比效果
- FR16: 用户可以放弃当前预览返回调整

**处理设置 (FR17-FR19):**
- FR17: 用户可以选择处理方法（AI 修复、模糊、填充、裁切）
- FR18: 用户可以调整处理方法的相关参数
- FR19: 用户可以重置当前区域设置到默认值

**批量处理 (FR20-FR26):**
- FR20: 用户可以启动批量处理任务
- FR21: 用户可以选择输出目录
- FR22: 系统显示批量处理进度
- FR23: 系统显示批量处理完成统计（成功/失败数量）
- FR24: 用户可以取消正在进行的批量任务
- FR25: 系统支持仅重试失败的项目
- FR26: 批量处理完成后系统提示用户保存模板

**模板管理 (FR27-FR34):**
- FR27: 用户可以保存当前处理设置为模板
- FR28: 用户可以为模板命名
- FR29: 用户可以查看已保存的模板列表
- FR30: 用户可以应用已保存的模板到当前任务
- FR31: 当应用模板到已有任务时，系统让用户选择应用方式
- FR32: 用户可以编辑已保存的模板
- FR33: 用户可以删除已保存的模板
- FR34: 系统显示最近使用的模板

**历史记录 (FR35-FR39):**
- FR35: 系统记录每次批量处理任务
- FR36: 用户可以查看历史任务列表
- FR37: 历史记录显示任务的基本信息（时间、图片数量、使用模板等）
- FR38: 用户可以删除单条历史记录
- FR39: 用户可以清空所有历史记录

**错误恢复 (FR40-FR42):**
- FR40: 用户可以清空当前任务返回首页
- FR41: 当有未保存内容时离开页面，系统提示用户确认
- FR42: 预览后用户可以放弃预览返回调整

**系统设置 (FR43-FR44):**
- FR43: 用户可以设置默认输出目录
- FR44: 用户可以设置输出文件命名规则

**模型加载 (FR45-FR46):**
- FR45: 系统在应用启动时在后台预加载 AI 模型
- FR46: 系统在模型加载完成后向用户显示就绪状态

### NonFunctional Requirements

**Performance (NFR1-NFR6):**
- NFR1: LaMa 模型加载完成后，预览生成应在 5 秒内完成（1920x1080 图片）
- NFR2: 应用启动时模型在后台预加载，不影响启动响应时间
- NFR3: 应用启动响应时间在 2 秒内
- NFR4: 批量处理进度实时更新，至少每秒刷新一次
- NFR5: 支持同时处理至少 100 张图片而无性能下降
- NFR6: 内存占用保持在 2GB 以下（处理 100 张 1920x1080 图片时）

**Security (NFR7-NFR11):**
- NFR7: 所有图片处理在本地完成，不上传任何数据到云端
- NFR8: 应用不收集用户使用数据
- NFR9: 模板和历史记录存储在本地用户目录
- NFR10: 应用仅访问用户明确选择的文件和目录
- NFR11: 处理结果仅写入用户指定的输出目录

**Reliability (NFR12-NFR16):**
- NFR12: 批量处理成功率 100%（用户错误如磁盘空间不足除外）
- NFR13: 单张图片处理失败不影响其他图片的处理
- NFR14: 批量任务可被用户取消，已处理的结果保留
- NFR15: 应用在处理异常情况下提供明确的错误信息
- NFR16: 应用在意外关闭后能恢复未保存的编辑状态

**Accessibility (NFR17-NFR19):**
- NFR17: 所有交互元素支持键盘导航
- NFR18: 界面元素具有清晰的标签和说明
- NFR19: 支持系统级缩放设置

### Additional Requirements (from Architecture)

**技术栈:**
- 前端：React 19 + TypeScript + Tailwind CSS
- 状态管理：Zustand（单一 Store + slice 模式）
- 桌面框架：Tauri 2.x
- 后端：Rust
- AI 模型：LaMa ONNX（本地运行）

**架构决策:**
- Zustand Store 结构：单一 Store + slice
- 响应式断点系统：Tauri 原生监听 + Hook
- Modal/Dialog 基础组件：自建 + Radix UI 原语
- 撤销中间件：Zustand 自定义 undo/redo
- 本地存储：SQLite (Tauri Plugin)

**Starter Template:**
- create-tauri-app 官方模板
- 命令：`npm create tauri-app@latest`

**平台打包:**
- macOS：DMG 打包 + 代码签名 + 公证
- Windows：安装程序
- Linux：AppImage、deb 包

### UX Design Requirements

**设计系统:**
- 颜色对比度：正文 4.5:1、大号 3:1、UI 组件 3:1
- 5 级窗口断点：Compact(1024px)、Narrow(1280px)、Normal(1440px)、Wide(1920px)、Ultrawide(2560px)
- 最小窗口尺寸：960px × 640px

**无障碍要求 (WCAG 2.1 AA):**
- 焦点陷阱组件：确认对话框、模板选择器、设置面板、预览模态
- ARIA 属性：aria-live、aria-describedby、aria-label、aria-modal、aria-busy
- 键盘快捷键：Cmd/Ctrl + O(打开)、Cmd/Ctrl + S(保存)、Cmd/Ctrl + B(侧边栏)、Esc(关闭模态)
- 跳过导航链接：桌面应用不需要

**UX 一致性模式:**
- 按钮层次：Primary/Secondary/Tertiary/Danger
- 反馈模式：错误带重试按钮
- 确认对话框：跨平台按钮位置 + 撤销机制
- 空状态：简化单按钮
- 加载状态：脉动动画 + 分段反馈
- 表单模式：字符计数 + 快捷键

**失败恢复机制:**
- 撤销窗口：8 秒撤销机会
- 检查点：定期保存状态
- 错误引导：每个错误带解决方案

**性能感知设计:**
- 骨架屏：模型加载时显示
- 分段进度反馈：分阶段消息（初始化、处理中、即将完成）
- 动态剩余时间估算
- 当前处理图片缩略图显示

## Epic List

### Epic 1: 图片导入与管理

**Epic Goal:**
用户可以导入图片文件或文件夹，系统显示缩略图列表，用户可以移除不需要的图片，系统自动选择第一张作为样图。

**FRs covered:** FR1, FR2, FR3, FR4, FR5

**User Outcome:**
"我可以快速导入一批图片，看到它们的缩略图，移除不需要的，然后开始处理第一张。"

### Epic 2: 处理区域配置

**Epic Goal:**
用户可以在图片上配置处理区域，包括拖动、缩放、旋转、清除操作，系统支持锚定模式和相对坐标模式。

**FRs covered:** FR6, FR7, FR8, FR9, FR10, FR11, FR12, FR13, FR17, FR18, FR19

**User Outcome:**
"我可以框选要处理的区域，调整大小和位置，选择处理方法，预览效果。"

### Epic 3: 预览与效果验证

**Epic Goal:**
用户可以生成当前区域和处理方式的预览效果，查看处理前后的对比，并可以放弃预览返回调整。

**FRs covered:** FR14, FR15, FR16

**User Outcome:**
"我可以在批量处理前预览效果，确保满意后再开始。"

### Epic 4: 批量处理执行

**Epic Goal:**
用户可以启动批量处理任务，选择输出目录，实时查看处理进度，查看完成统计，并支持取消和重试失败项。

**FRs covered:** FR20, FR21, FR22, FR23, FR24, FR25, FR26

**User Outcome:**
"我可以一键批量处理所有图片，看到实时进度，完成后知道成功和失败的数量。"

### Epic 5: 模板与历史管理

**Epic Goal:**
用户可以保存当前处理设置为模板，为模板命名，查看和应用模板，编辑和删除模板。系统还记录处理历史，用户可以查看和管理历史记录。

**FRs covered:** FR27, FR28, FR29, FR30, FR31, FR32, FR33, FR34, FR35, FR36, FR37, FR38, FR39

**User Outcome:**
"我可以保存常用配置为模板，下次直接应用，还能查看之前的处理记录。"

### Epic 6: 系统基础与设置

**Epic Goal:**
系统在后台预加载 AI 模型，加载完成后显示就绪状态。用户可以设置默认输出目录和文件命名规则。系统支持错误恢复机制，包括清空任务确认和离开页面确认。

**FRs covered:** FR40, FR41, FR42, FR43, FR44, FR45, FR46

**User Outcome:**
"应用启动很快，模型在后台加载，我可以配置默认设置，不用担心误操作，系统会让我确认。"

## Epic Details

### Epic 1: 图片导入与管理

#### Story 1.1: 导入图片

**作为**普通用户，
**我想要**导入图片文件或文件夹，
**这样**我就可以开始批量处理。

**FRs covered:** FR1, FR2

**UX Design Requirements:**
- 按钮层次：Primary 按钮"导入图片"和"导入文件夹"
- 反馈模式：成功时显示"已导入 X 张图片"提示
- 键盘快捷键：Cmd/Ctrl + O 打开文件选择器

**Acceptance Criteria:**

**Scenario 1: 导入单个或多个图片文件**

**Given** 用户在首页
**When** 用户点击"导入图片"按钮并选择单个或多个图片文件
**Then** 系统将选中的图片添加到图片列表
**And** 系统显示"已导入 X 张图片"的提示消息
**And** 系统在列表中显示第一张图片的预览

**Scenario 2: 导入包含图片的文件夹**

**Given** 用户在首页
**When** 用户点击"导入文件夹"按钮并选择包含图片的文件夹
**Then** 系统扫描文件夹并将所有支持的图片添加到图片列表
**And** 系统显示"从文件夹导入 X 张图片"的提示消息
**And** 系统在列表中显示第一张图片的预览

**Scenario 3: 导入不支持的文件格式**

**Given** 用户选择了不支持的文件格式
**When** 用户尝试导入
**Then** 系统显示错误提示"不支持的文件格式，请选择 JPG、PNG、WEBP 等图片格式"
**And** 系统不添加该文件到列表

**Scenario 4: 键盘快捷键导入**

**Given** 用户在首页
**When** 用户按下 Cmd/Ctrl + O
**Then** 系统打开文件选择器
**And** 用户选择文件后按 Scenario 1 执行

**Technical Notes:**
- 使用 Tauri 的文件选择 API
- 支持的格式：JPG、JPEG、PNG、WEBP
- 文件夹扫描需要递归读取子目录

#### Story 1.2: 图片列表管理

**作为**普通用户，
**我想要**查看已导入图片的缩略图列表并移除不需要的图片，
**这样**我就可以只处理需要的图片。

**FRs covered:** FR3, FR4, FR5

**UX Design Requirements:**
- 缩略图列表：按文件名排序，当前选中项有视觉高亮
- 空状态模式：显示"还没有导入图片，点击上方按钮导入"
- 键盘导航：方向键切换图片，Delete/Backspace 移除

**Acceptance Criteria:**

**Scenario 1: 显示图片缩略图列表**

**Given** 用户已导入至少一张图片
**When** 系统显示图片列表
**Then** 列表显示每张图片的缩略图
**And** 缩略图按文件名排序显示
**And** 当前选中的图片有视觉高亮边框

**Scenario 2: 切换当前选中图片**

**Given** 当前有选中的图片
**When** 用户点击某张图片的缩略图
**Then** 该图片成为当前选中图片
**And** 主预览区域更新显示该图片
**And** 区域框工具栏激活

**Scenario 3: 移除单张图片**

**Given** 图片列表中有不需要的图片
**When** 用户点击图片缩略图右上角的"×"按钮
**Then** 该图片从列表中移除
**And** 系统显示"已移除 [文件名]"的提示消息
**And** 如果移除的是当前选中图片，系统自动选中下一张图片（如果存在）

**Scenario 4: 自动选择第一张作为样图**

**Given** 用户导入图片后
**When** 图片列表显示
**Then** 系统自动选择第一张图片作为样图
**And** 样图在主预览区域显示
**And** 样图有视觉指示（蓝色边框或高亮）

**Scenario 5: 空状态显示**

**Given** 图片列表为空
**When** 用户查看图片列表区域
**Then** 系统显示空状态提示"还没有导入图片，点击上方按钮导入"
**And** 显示"导入图片"按钮

**Scenario 6: 键盘导航**

**Given** 图片列表中有图片
**When** 用户按上下方向键
**Then** 选中的图片在列表中上下移动
**And** 主预览区域同步更新

**Given** 有选中的图片
**When** 用户按下 Delete 或 Backspace 键
**Then** 当前选中的图片被移除
**And** 按 Scenario 3 执行移除逻辑

**Technical Notes:**
- 缩略图使用懒加载优化性能
- 使用 Zustand slice 管理图片列表状态
- 键盘事件监听需要焦点管理

### Epic 2: 处理区域配置

#### Story 2.1: 默认区域框显示

**作为**普通用户，
**我想要**在图片上看到默认的处理区域框，
**这样**我就可以知道在哪里进行处理，或者直接调整使用。

**FRs covered:** FR6

**UX Design Requirements:**
- 默认框位置：图片右下角，典型水印位置
- 视觉反馈：虚线边框，半透明填充，角上手柄
- 无障碍：区域框有 aria-label 描述

**Acceptance Criteria:**

**Scenario 1: 显示默认区域框**

**Given** 用户选中了一张图片
**When** 图片在主预览区域显示
**Then** 系统在图片上显示一个默认的处理区域框
**And** 默认框位于图片右下角
**And** 默认框大小为图片宽高的 20% × 10%
**And** 框有虚线边框和半透明填充

**Scenario 2: 默认框视觉样式**

**Given** 默认区域框显示在图片上
**When** 用户查看区域框
**Then** 框有四个角的手柄（圆形可拖拽点）
**And** 边框颜色与图片形成对比
**And** 悬停时手柄放大显示

**Scenario 3: 无障碍支持**

**Given** 区域框显示在图片上
**When** 屏幕阅读器读取
**Then** 区域框有 aria-label="处理区域框，可拖动移动或使用手柄调整大小"

**Technical Notes:**
- 使用 CSS absolute 定位实现区域框
- 区域框坐标使用相对百分比（0-100）以适应不同图片尺寸
- 默认位置：x=80, y=90, width=20, height=10（百分比）

#### Story 2.2: 区域框拖动与缩放

**作为**普通用户，
**我想要**拖动区域框改变位置，拖拽手柄调整大小，
**这样**我就可以精确框选要处理的区域。

**FRs covered:** FR7, FR8

**UX Design Requirements:**
- 鼠标交互：拖动框体移动位置，拖动手柄缩放
- 触觉反馈：光标变化（move、nwse-resize 等）
- 边界约束：框不能完全移出图片

**Acceptance Criteria:**

**Scenario 1: 拖动区域框改变位置**

**Given** 图片上显示有区域框
**When** 用户点击区域框内部并拖动
**Then** 区域框跟随鼠标移动
**And** 拖动时鼠标光标变为 move 样式
**And** 区域框保持在图片边界内

**Scenario 2: 拖拽手柄调整大小**

**Given** 图片上显示有区域框
**When** 用户点击某个角的手柄并拖动
**Then** 区域框从该角开始调整大小
**And** 拖动时鼠标光标变为对应的 resize 样式
**And** 实时显示当前框的尺寸（像素或百分比）

**Scenario 3: 边界约束**

**Given** 用户正在拖动区域框
**When** 框的边缘到达图片边界
**Then** 框不能超出图片边界
**And** 框的最小尺寸为 10×10 像素

**Scenario 4: 四个角手柄独立工作**

**Given** 区域框显示中
**When** 用户分别拖动左上、右上、左下、右下手柄
**Then** 每个手柄独立控制对应角的调整
**And** 对角的手柄保持固定

**Technical Notes:**
- 使用鼠标事件监听实现拖拽（mousedown, mousemove, mouseup）

**Then** 系统显示骨架屏启动页
**And** 显示应用名称和 Logo

**Then** 系统在后台开始加载 LaMa ONNX 模型
**And** 加载进度不阻塞用户操作

**Scenario 3: 显示加载进度**

- 模型加载使用 Tauri async command
- 通过 Tauri events 向前端推送加载状态
- 模型文件打包在应用资源中

#### Story 6.3: 错误恢复机制

**作为**普通用户，
**我想要**在误操作时有确认提示，可以清空当前任务或离开页面时得到保护，
**这样**我就不会因为不小心而丢失工作。

**FRs covered:** FR40, FR41, FR42

**UX Design Requirements:**
- 确认对话框：标准三选项或二选项模式
- 撤销窗口：8 秒撤销机会
- 脏状态检测：检测未保存的编辑内容

**Acceptance Criteria:**

**Scenario 1: 清空当前任务确认**

**Given** 用户在模板构建页，已导入图片并进行了调整
**When** 用户点击"清空当前任务"按钮
**Then** 系统显示确认对话框"清空当前任务？"
**And** 对话框显示警告"清空后将移除当前导入的图片、预览结果和未保存的编辑内容，并返回首页。"
**And** 对话框有"清空"和"取消"按钮
**And** "清空"按钮使用 Danger 样式

**Given** 用户点击"清空"
**When** 系统执行清空操作
**Then** 所有图片从列表中移除
**And** 预览结果被清除
**And** 区域框重置为默认状态
**And** 返回首页
**And** 提供 8 秒撤销窗口"任务已清空 [撤销]"

**Scenario 2: 离开页面确认**

**Given** 用户在模板构建页，有未保存的编辑内容（调整了区域框但未预览）
**When** 用户尝试关闭应用窗口
**Then** 系统显示确认对话框"确定要离开吗？您有未保存的更改。"
**And** 对话框有"离开"和"留下"按钮
**And** 点击"留下"后应用不关闭
**And** 点击"离开"后应用关闭，更改丢失

**Scenario 3: 预览后返回确认**

**Given** 用户在预览页面
**When** 用户点击"返回调整"按钮
**Then** 系统显示确认提示"预览将被放弃，返回调整区域？"
**And** 有"确认"和"取消"按钮
**And** 点击"确认"后返回模板构建页
**And** 预览结果被清除
**And** 区域框保持预览前的状态

**Scenario 4: 脏状态检测**

**Given** 用户在模板构建页
**When** 用户调整区域框位置或大小
**Then** 系统标记状态为"有未保存更改"
**And** 尝试离开时触发 Scenario 2 的确认对话框

**Given** 用户保存了模板
**When** 用户尝试离开页面
**Then** 系统不显示确认对话框
**And** 应用正常关闭

**Scenario 5: 批量处理中离开确认**

**Given** 批量处理正在进行中
**When** 用户尝试关闭应用窗口
**Then** 系统显示确认对话框"批量处理正在进行中，确定要中断并退出吗？"
**And** 对话框警告"已处理的结果将保留，未处理的需要重新处理。"
**And** 有"退出"和"取消"按钮
**And** 点击"退出"后应用停止处理并关闭

**Technical Notes:**
- 使用 Zustand 的 dirty state tracking
- Tauri window close event 监听
- 撤销功能使用 toast + 定时器实现
