# Product Requirements Document - Batch Image Studio

**Author:** Luoyaosheng
**Date:** 2026-03-27

---

## Executive Summary

Batch Image Studio 是一款面向普通个人用户的跨平台桌面图片批量处理工具，专注于简化水印清理流程。

**产品愿景**：让不懂 Photoshop 的普通用户也能快速完成批量图片清理工作。用户只需框选处理区域、预览效果、一键批量，无需学习复杂的专业工具。

**核心问题**：现有解决方案存在两极分化——专业工具（如 Photoshop）学习成本过高，在线工具存在隐私风险且功能受限。普通用户需要一个既能保护隐私、又简单易用的本地批量处理方案。

**目标用户**：需要处理 AI 生成图片水印的普通个人用户，主要场景是批量去除豆包等 AI 工具生成图片的角标水印。

---

### What Makes This Special

**核心差异**：Batch Image Studio 不是另一款通用图片编辑器，而是"简化版 PS 批量动作"——将 Photoshop 中"新建选区 → 选择修复工具 → 录制动作 → 批量运行"的复杂流程，简化为"导入图片 → 框选区域 → 预览 → 批量处理"。

**模板系统**：用户的一次处理设置可以保存为模板，下次直接复用。模板记住一切设置（区域位置、处理方法、参数），无需重复配置。

**本地优先**：所有处理在本地完成，使用 LaMa ONNX 模型进行 AI 修复，无需上传图片到云端，保护用户隐私。

**技术栈**：Tauri 2.x + React 19 + Rust，跨平台支持 macOS/Windows/Linux。

## Success Criteria

### User Success

用户成功使用产品的关键指标：

- **完整流程可达性**：用户能够独立完成"导入图片 → 框选区域 → 预览效果 → 批量处理 → 导出结果"的完整流程，无需查阅文档或教程
- **模板复用效率**：用户保存模板后，下次使用时可直接应用，无需重新配置参数
- **错误恢复能力**：用户在任何步骤都能轻松返回或撤销，不会因为误操作而需要重启应用
- **零学习曲线**：首次使用用户能在 5 分钟内完成第一次批量处理任务

### Business Success

作为开源项目的成功指标：

- **社区采用**：项目在 GitHub 获得 Star 和 Fork，开发者社区贡献代码或反馈问题
- **用户留存**：用户下载后能成功运行并完成至少一次批量处理任务
- **问题解决**：用户提出的 Issue 能在合理时间内得到响应和修复
- **跨平台兼容**：应用在 macOS、Windows、Linux 三个平台都能正常运行

### Technical Success

技术层面的成功要求：

- **批量处理成功率**：100% — 对于支持的图片格式，批量处理不应因应用本身问题而失败（用户错误如磁盘空间不足除外）
- **预览生成响应**：LaMa ONNX 模型加载完成后，预览生成应在模型能力范围内尽可能快
- **模型加载体验**：模型在后台预加载，完成后给予用户明确的"就绪"提示，不影响启动体验
- **内存稳定性**：批量处理大尺寸或多张图片时不发生内存溢出或崩溃
- **状态一致性**：所有清除/撤销/模板切换操作后，应用状态保持一致，无数据丢失或错乱

### Measurable Outcomes

可量化的结果指标：

- **交互流程完整性**：补充稿中定义的 6 个必须入口全部实现
- **工作流时序覆盖**：9 个核心工作流时序图对应的交互全部正确实现
- **模板功能**：用户能保存、应用、编辑、删除模板
- **本地 AI 处理**：LaMa ONNX 模型能在本地环境成功运行
- **跨平台打包**：能生成三个平台的安装包

## User Journeys

### Journey 1: 首次用户 - 小明的故事

**角色背景**：
小明是一名普通上班族，最近用豆包 AI 生成了 100 张配图用于工作汇报。但每张图右下角都有豆包的水印，他很苦恼。

**Opening Scene（开场）**：
小明尝试用在线去水印工具，但看到要上传图片就担心隐私问题。他听说有个叫 Batch Image Studio 的开源工具，可以本地处理，决定试试。

**Rising Action（行动过程）**：
1. 小明下载并安装应用，双击启动
2. 首页简洁明了，他看到"导入图片"按钮，点击选择文件夹
3. 100 张图片导入成功，显示缩略图列表
4. 第一张图自动显示在中间，右下角已经有一个默认的框
5. 他调整了一下框的大小，正好框住水印
6. 点击"预览效果"，3 秒后看到水印被完美清除
7. 系统提示："这次的设置可以保存为模板，下次直接使用。"小明点击"保存模板"，命名为"豆包水印"
8. 点击"开始批量处理"，选择输出目录
9. 进度条显示处理进度，2 分钟后全部完成
10. 系统提示："处理完成！100 张成功，0 张失败。"

**Climax（关键时刻）**：
小明打开输出目录，看到所有图片的水印都被完美清除。他长舒一口气——这下不用担心版权问题了。

**Resolution（结局）**：
小明很满意这个工具，决定推荐给同事。下次再有类似任务，他直接用保存的模板就能搞定。

---

### Journey 2: 重复用户 - 小华的故事

**角色背景**：
小华是一名自媒体创作者，经常需要批量处理 AI 生成的配图。她已经是 Batch Image Studio 的老用户了。

**Opening Scene（开场）**：
小华又有一批新图需要处理，她熟练地打开应用。

**Rising Action（行动过程）**：
1. 应用启动后，首页显示"最近模板"，"豆包水印"模板在最前面
2. 小华点击"应用模板"，系统弹出选择："应用到当前图片 / 清空任务后应用 / 取消"
3. 她选择"清空任务后应用"，但这次她选择"导入文件夹"而不是"导入图片"
4. 新的 50 张图片导入，默认框自动应用
5. 小华快速预览第一张，确认效果
6. 直接点击"开始批量处理"

**Climax（关键时刻）**：
50 张图在 1 分钟内处理完成。小华只需要选输出目录，其他都是自动的。

**Resolution（结局）**：
小华很满意效率，继续她的创作工作。这个工具已经成为了她的工作流中不可或缺的一部分。

### Journey 3: 错误恢复 - 小刚的误操作

**角色背景**：
小刚是第一次使用，不小心选错了图片，或者框选错了位置。

**Opening Scene（开场）**：
小刚导入了一批图片，但不小心把不该导入的文件夹也导入了。

**Rising Action（行动过程）**：
1. 小刚发现导入了错误的图片，有点慌张
2. 他看到左侧列表每张图片旁边有个小的"×"按钮
3. 他点了几下，把错误的图片移除了
4. 在调整框时，不小心把框拖到了图片外面
5. 他看到工具条上有"重新框选"按钮，点击后重新拖拽
6. 处理完成后，他想换一批图片处理
7. 他点击左侧的"清空当前任务"按钮
8. 系统弹出确认："清空当前任务？清空后将移除当前导入的图片、预览结果和未保存的编辑内容，并返回首页。"

**Climax（关键时刻）**：
小刚发现无论怎么误操作，都有"回来"的路。他放心地继续使用。

**Resolution（结局）**：
小刚成功完成了任务，对产品的容错性印象深刻。

## Desktop App Specific Requirements

### Project-Type Overview

Batch Image Studio 是一款跨平台桌面应用，基于 Tauri 2.x + React 19 + Rust 技术栈，支持 macOS、Windows 和 Linux 三个平台。应用专注于本地图片批量处理，使用 LaMa ONNX 模型进行 AI 修复，所有处理在本地完成，保护用户隐私。

### Technical Architecture Considerations

**技术栈**：
- **前端框架**：React 19 + TypeScript + Tailwind CSS
- **状态管理**：Zustand
- **桌面框架**：Tauri 2.x
- **后端逻辑**：Rust
- **AI 模型**：LaMa ONNX（本地运行）

**架构特点**：
- Tauri 提供原生窗口和系统集成
- React 处理 UI 和用户交互
- Rust 处理图片处理和 ONNX 模型调用
- 本地文件存储，无需云端依赖

### Platform Support

**目标平台**：
- macOS (Intel + Apple Silicon)
- Windows (x64)
- Linux (x64)

**平台特定考虑**：
- **macOS**：代码签名、公证、DMG 安装包
- **Windows**：安装程序、可能需要 SmartScreen 警告处理
- **Linux**：AppImage、deb 包

**UI 一致性**：
- 跨平台保持一致的 UI 体验
- 遵循各平台的原生 UI 规范（窗口控制、菜单栏等）

### System Integration

**文件系统集成**：
- 读取图片文件（拖拽导入、文件夹选择）
- 写入处理结果（输出目录选择）
- 模板和历史记录的本地存储

**系统集成点**：
- 文件关联（可选）：双击图片文件用本应用打开
- 右键菜单（可选）：在文件管理器中快速处理

### Update Strategy

**当前策略**：
- GitHub Releases 提供下载
- 用户手动检查更新

**未来考虑**：
- Tauri 内置更新器
- 启动时自动检查更新提示

### Offline Capabilities

**完全离线运行**：
- 所有功能本地处理，无需网络连接
- LaMa ONNX 模型打包在应用中
- 无需账号或云端服务

**数据存储**：
- 模板存储在本地用户目录
- 历史记录存储在本地
- 无云端同步

### Implementation Considerations

**内存管理**：
- 大图片处理时的内存控制
- 批量处理时的内存释放
- ONNX 模型的内存占用

**性能优化**：
- 模型预加载和缓存
- 批量处理的并行处理
- UI 线程与处理线程分离

**跨平台构建**：
- CI/CD 自动构建三个平台的安装包
- 代码签名配置
- 平台特定测试

- FR46: 系统在模型加载完成后向用户显示就绪状态

## Non-Functional Requirements

### Performance

**预览生成性能**：
- NFR1: LaMa 模型加载完成后，预览生成应在 5 秒内完成（1920x1080 图片）
- NFR2: 应用启动时模型在后台预加载，不影响启动响应时间
- NFR3: 应用启动响应时间在 2 秒内

**批量处理性能**：
- NFR4: 批量处理进度实时更新，至少每秒刷新一次
- NFR5: 支持同时处理至少 100 张图片而无性能下降
- NFR6: 内存占用保持在 2GB 以下（处理 100 张 1920x1080 图片时）

### Security

**数据隐私**：
- NFR7: 所有图片处理在本地完成，不上传任何数据到云端
- NFR8: 应用不收集用户使用数据
- NFR9: 模板和历史记录存储在本地用户目录

**文件访问**：
- NFR10: 应用仅访问用户明确选择的文件和目录
- NFR11: 处理结果仅写入用户指定的输出目录

### Reliability

**批量处理稳定性**：
- NFR12: 批量处理成功率 100%（用户错误如磁盘空间不足除外）
- NFR13: 单张图片处理失败不影响其他图片的处理
- NFR14: 批量任务可被用户取消，已处理的结果保留

**错误处理**：
- NFR15: 应用在处理异常情况下提供明确的错误信息
- NFR16: 应用在意外关闭后能恢复未保存的编辑状态

### Accessibility

**基本无障碍支持**：
- NFR17: 所有交互元素支持键盘导航
- NFR18: 界面元素具有清晰的标签和说明
- NFR19: 支持系统级缩放设置
